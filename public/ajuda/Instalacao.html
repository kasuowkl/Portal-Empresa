Instalação e Configuração — Linux
Guia completo para implantar o Portal WKL em um servidor Ubuntu/Debian. Tempo estimado: 20–30 minutos.
Guia testado em Ubuntu 22.04 LTS e Debian 12. O banco de dados SQL Server pode estar em outro servidor na rede — não precisa estar no mesmo host.

Pré-requisitos

O servidor Linux precisa ter acesso de rede ao SQL Server e as seguintes ferramentas instaladas:

Node.js
v18 LTS ou superior
npm
v9+ (vem com o Node.js)
Nginx
Opcional — proxy reverso
PM2
Gerenciador de processo Node
SQL Server
2017+ (pode ser remoto)
ufw / iptables
Firewall do servidor
Execute os comandos como root ou prefixe com sudo. Recomendado criar um usuário dedicado (ex: portaluser) para rodar a aplicação.

1 Instalar Node.js 18 LTS

Use o repositório oficial da NodeSource para instalar a versão LTS mais recente:

bash — Ubuntu / Debian
# Atualiza os pacotes do sistema
apt update && apt upgrade -y

# Instala dependências necessárias
apt install -y curl ca-certificates gnupg

# Adiciona o repositório NodeSource (Node 18 LTS)
curl -fsSL https://deb.nodesource.com/setup_18.x | bash -

# Instala o Node.js e o npm
apt install -y nodejs

# Verifica as versões instaladas
node --version   # Deve exibir v18.x.x ou superior
npm --version    # Deve exibir 9.x.x ou superior

Instale o PM2 globalmente (gerenciador de processos Node.js):

bash
npm install -g pm2

2 Estrutura de Arquivos

Crie o diretório de instalação e ajuste as permissões:

bash
# Cria o diretório de instalação
mkdir -p /var/www/html/wkl/Portal
cd /var/www/html/wkl/Portal

# Cria subdiretórios de logs (necessários na primeira execução)
mkdir -p logs uploads

# Ajusta permissões (adapte o usuário conforme seu ambiente)
chown -R www-data:www-data /var/www/html/wkl/Portal
chmod -R 755 /var/www/html/wkl/Portal

Copie os arquivos do Portal do seu computador Windows para o servidor via SCP:

powershell — executar no Windows
# Copia todos os arquivos do portal para o servidor
scp -r "C:\Users\kasuo\Desktop\Portal\*" user@192.168.0.80:/var/www/html/wkl/Portal/
Não copie a pasta node_modules/ nem o arquivo .env pelo SCP. Os módulos serão instalados via npm install no servidor e o .env deve ser criado manualmente.

4 Variáveis de Ambiente (.env)

Crie o arquivo .env na raiz do projeto com as configurações do seu ambiente:

bash — criar o arquivo
nano /var/www/html/wkl/Portal/.env
.env — conteúdo do arquivo
# ============================================================
# Banco de Dados (SQL Server)
# ============================================================
DB_SERVER=192.168.0.209        # IP ou hostname do SQL Server
DB_PORT=1433                   # Porta padrão do SQL Server
DB_USER=sa                     # Usuário do banco
DB_PASSWORD=SuaSenhaAqui       # Senha do banco
DB_NAME=PortalSistemas         # Nome do banco de dados

# ============================================================
# Servidor Web
# ============================================================
PORTA_APP=3132                 # Porta em que o portal vai escutar

# ============================================================
# Sessão (use uma string longa e aleatória)
# ============================================================
SESSION_SECRET=troque_por_texto_secreto_longo_aqui_2026

# ============================================================
# Criptografia — Senhas de App do Google Calendar (CalDAV)
# ============================================================
ENCRYPTION_KEY=troque_por_chave_de_32_bytes_aqui!  # Exatamente 32 caracteres
Nunca versione o arquivo .env nem o compartilhe. Ele contém credenciais sensíveis. Adicione ao .gitignore se usar Git: echo ".env" >> .gitignore

Usuário: admin  |  Senha: admin

Verificar conectividade com o SQL Server:

bash — teste de porta TCP
# Testa se a porta 1433 do SQL Server está acessível
nc -zv 192.168.0.209 1433

# Alternativa com telnet
telnet 192.168.0.209 1433

Se a conexão falhar, verifique:

  • O SQL Server está com TCP/IP habilitado no SQL Server Configuration Manager.
  • O Firewall do Windows permite conexões na porta 1433.
  • As credenciais no .env estão corretas e o usuário tem permissão de dbcreator.

6 PM2 — Gerenciador de Processo

O PM2 mantém o portal em execução permanente, reinicia automaticamente em caso de crash e inicia com o sistema operacional.

Iniciar a aplicação:

bash
cd /var/www/html/wkl/Portal

# Inicia o portal com PM2
pm2 start portal.js --name portal

# Verifica se está rodando
pm2 status

Configurar inicialização automática no boot:

bash
# Gera o comando de startup (copie e execute o comando gerado)
pm2 startup

# Salva a lista de processos atual para restaurar no boot
pm2 save
O comando pm2 startup exibe um comando personalizado para o seu sistema. Copie e execute esse comando como root para registrar o serviço.

Comandos úteis do PM2:

bash — gerenciamento do processo
# Ver status de todos os processos
pm2 status

# Ver logs em tempo real
pm2 logs portal

# Reiniciar o portal (após atualização de arquivos)
pm2 restart portal

# Parar o portal
pm2 stop portal

# Monitorar CPU e memória em tempo real
pm2 monit

# Ver informações detalhadas
pm2 show portal

7 Nginx — Proxy Reverso

O Nginx recebe as requisições na porta 80 (HTTP padrão) e as repassa ao Node.js na porta 3132. Isso permite acessar o portal sem digitar a porta na URL.

1
Instalar o Nginx:
bash
apt install -y nginx
systemctl enable nginx
systemctl start nginx
2
Criar o arquivo de configuração do site:
bash
nano /etc/nginx/sites-available/portal-wkl
3
Conteúdo do arquivo de configuração:
nginx — /etc/nginx/sites-available/portal-wkl
server {
    listen 80;
    server_name 192.168.0.80;  # IP ou domínio do servidor

    # Tamanho máximo de upload (para Restore de banco)
    client_max_body_size 60m;

    # Proxy para o Node.js
    location / {
        proxy_pass         http://127.0.0.1:3132;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_cache_bypass $http_upgrade;

        # Timeout para operações longas (backup/restore)
        proxy_read_timeout 120s;
        proxy_connect_timeout 10s;
    }

    # Logs de acesso e erro
    access_log /var/log/nginx/portal-wkl.access.log;
    error_log  /var/log/nginx/portal-wkl.error.log;
}
4
Ativar o site e recarregar o Nginx:
bash
# Cria o link simbólico para ativar o site
ln -s /etc/nginx/sites-available/portal-wkl /etc/nginx/sites-enabled/

# Remove o site padrão (opcional)
rm -f /etc/nginx/sites-enabled/default

# Valida a configuração
nginx -t

# Recarrega o Nginx
systemctl reload nginx
Após configurar o Nginx, o portal fica acessível em http://192.168.0.80/ (sem porta). Se preferir manter o acesso direto pela porta 3132, o Nginx é opcional.

8 Firewall (ufw)

Libere apenas as portas necessárias no firewall do servidor Linux:

bash — configuração do ufw
# Ativa o ufw (confirme com "y")
ufw enable

# Permite SSH (mantenha sempre liberado!)
ufw allow 22/tcp

# Opção A: usando Nginx na porta 80 (recomendado)
ufw allow 80/tcp

# Opção B: acesso direto ao Node.js (sem Nginx)
ufw allow 3132/tcp

# Verifica as regras ativas
ufw status verbose
Nunca bloqueie a porta 22 (SSH) antes de verificar que consegue acessar o servidor novamente — caso contrário, ficará sem acesso remoto.

Acesso ao SQL Server

A porta 1433 do SQL Server (em 192.168.0.209) deve estar liberada no firewall do servidor Windows para receber conexões do IP do servidor Linux. Configure uma regra de entrada no Windows Firewall ou Windows Defender Firewall.

10 Atualização do Portal

Para aplicar uma nova versão dos arquivos sem interromper o serviço por mais de alguns segundos:

bash — deploy de atualização
# 1. Copie os arquivos novos do Windows para o servidor (via SCP)
#    Exemplo — copiar apenas os arquivos modificados:
#    scp "C:\Portal\routes\portal.js" user@192.168.0.80:/var/www/html/wkl/Portal/routes/

# 2. Se adicionou novas tabelas, rode o setup do banco
cd /var/www/html/wkl/Portal
node criarBancoPortal.js

# 3. Reinicia o processo (zero-downtime)
pm2 restart portal

# 4. Verifica se voltou a funcionar
pm2 status
pm2 logs portal --lines 20
Para atualizações que envolvem muitos arquivos, use scp -r para copiar toda a pasta de uma vez e depois pm2 restart portal.

Script de deploy completo (opcional):

bash — salvar como /usr/local/bin/deploy-portal

# Script de deploy do Portal WKL
set -e

APP_DIR="/var/www/html/wkl/Portal"

deploy-portal

11 Sincronização Google Calendar (CalDAV)

O Portal WKL integra com o Google Calendar através do protocolo CalDAV usando autenticação por Senha de App. Não é necessário Google Cloud Console, OAuth2 ou registro de aplicativo — apenas um e-mail Google e uma senha de app gerada no painel da conta.

A integração é opcional e configurada individualmente por usuário no módulo Calendários. Nenhuma configuração extra no servidor é necessária além da variável ENCRYPTION_KEY no .env.

Pré-requisito no servidor: ENCRYPTION_KEY

As senhas de app são armazenadas criptografadas (AES-256-CBC) no banco de dados. Para isso funcionar, defina a variável no .env:

.env
# Chave de criptografia — deve ter EXATAMENTE 32 caracteres
ENCRYPTION_KEY=MinhaChaveSecreta32Chars2026!!

Para gerar uma chave segura via terminal:

bash
# Gera 32 bytes aleatórios em base64 (use os primeiros 32 caracteres)
openssl rand -base64 32 | head -c 32
Se a ENCRYPTION_KEY for alterada após usuários já terem configurado o CalDAV, as senhas salvas não poderão mais ser descriptografadas e os usuários precisarão reconfigurar a integração.

Configuração por usuário (no Portal)

Cada usuário configura sua própria integração na sidebar do módulo Calendários:

1
Acesse myaccount.google.com/apppasswords com a conta Google.
2
Crie uma Senha de App para "Calendário" ou "Outro". Copie a senha gerada (16 caracteres sem espaços).
3
No Portal → Calendários → sidebar → clique em Conectar Google Calendar.
4
Informe o e-mail Google e a Senha de App e clique em Conectar.
As Senhas de App só estão disponíveis em contas Google com verificação em duas etapas ativada. Se a opção não aparecer em myaccount.google.com/apppasswords, ative a verificação em duas etapas primeiro.

Como funciona a sincronização

Portal → Google

Ao criar, editar ou excluir um evento no Portal, a alteração é enviada automaticamente para o Google Calendar via CalDAV.

Google → Portal

O cron cronCalendarios.js sincroniza eventos do Google para o Portal a cada 30 minutos. O usuário também pode acionar a sync manualmente pelo botão Sincronizar Agora na sidebar.

Segurança

A senha de app é armazenada criptografada com AES-256-CBC usando a ENCRYPTION_KEY do .env. Nunca é exibida em texto claro após salva.

Verificar o cron de sincronização

bash — verificar logs do cron
# Ver logs do portal (o cron registra cada ciclo)
pm2 logs portal --lines 50 | grep -i "caldav\|calendarios\|sync"

# Ou monitorar em tempo real
pm2 logs portal | grep -i "calendar"

O cron executa dois jobs automaticamente:

  • Lembretes — a cada hora, envia e-mails para eventos nas próximas 24h
  • Sync Google → Portal — a cada 30 minutos, importa novos eventos/alterações do Google

Solução de Problemas

Erro: "Cannot connect to SQL Server"

1. Verifique a conectividade: nc -zv 192.168.0.209 1433
2. Confirme que TCP/IP está habilitado no SQL Server (SQL Server Configuration Manager → Protocols for MSSQLSERVER → TCP/IP = Enabled).
3. Verifique as credenciais no .env — o usuário precisa ter permissão dbcreator ou acesso ao banco PortalSistemas.
4. Confirme que o Firewall do Windows libera a porta 1433 para o IP do servidor Linux.

Erro: "EADDRINUSE — porta 3132 em uso"

Outra instância do portal já está rodando. Pare ela primeiro:
pm2 stop portal ou fuser -k 3132/tcp

Portal não inicia após reboot

Execute novamente os comandos de startup do PM2:
pm2 startup → copie e execute o comando gerado → pm2 save

Nginx retorna 502 Bad Gateway

O Node.js não está rodando ou está na porta errada.
1. pm2 status — verifique se o processo está online.
2. pm2 restart portal — reinicia o processo.
3. Confirme que a porta no .env (PORTA_APP) bate com o proxy_pass do Nginx.

Login não funciona (sessão não persiste)

Verifique se o SESSION_SECRET no .env está definido e não está em branco. Após alterar o .env, sempre reinicie: pm2 restart portal.

CalDAV: "Erro ao conectar com Google Calendar"

1. Confirme que a conta Google tem verificação em duas etapas ativada.
2. A senha de app deve ter 16 caracteres — remova espaços se copiou com espaços do Google.
3. Verifique se o servidor tem acesso HTTPS à internet (curl -I https://apidata.googleusercontent.com).
4. Confirme que a ENCRYPTION_KEY no .env tem exatamente 32 caracteres.
5. Após alterar o .env, reinicie: pm2 restart portal.

Permissão negada em /logs ou /uploads

Ajuste o dono dos diretórios para o usuário que roda o PM2:
chown -R $USER:$USER /var/www/html/wkl/Portal/logs /var/www/html/wkl/Portal/uploads

Diagnóstico geral:

bash
# Checar todos os processos PM2
pm2 status

# Ver erros recentes da aplicação
pm2 logs portal --err --lines 30

# Testar se o portal responde localmente
curl -I http://127.0.0.1:3132/login.html

# Checar se a porta está escutando
ss -tlnp | grep 3132

# Checar uso de memória e CPU
pm2 monit
