Visão Geral do Projeto

O Portal WKL de Sistemas é uma aplicação web interna que serve como ponto central de acesso a todos os sistemas e serviços da empresa. Após autenticar-se, o usuário visualiza uma grade com todos os sistemas disponíveis e pode navegar para qualquer um deles com um clique.

O portal é acessado pela rede local no endereço http://192.168.0.80:3132 e suporta dois tipos de autenticação: usuários locais (banco de dados) e usuários do domínio Active Directory (LDAP). O módulo Agenda de Tarefas está integrado ao mesmo servidor, acessível em /agenda.

Objetivo principal

Centralizar o acesso a todos os sistemas internos da empresa com autenticação segura (local + AD), painel administrativo completo e módulos integrados como a Agenda de Tarefas.

5.2 Login via Active Directory (LDAP)

Quando o login local falha, o sistema verifica se o usuário está na tabela usuarios_dominio e, se estiver, autentica via LDAP usando as configurações salvas no banco (grupo ad).

Dados armazenados na sessão

// req.session.usuario
{
  id:      1,
  nome:    "João da Silva",
  usuario: "joao.silva",
  nivel:   "usuario",   // "admin" | "usuario"
  tipo:    "dominio"    // "local" | "dominio"
}

Configurações do Active Directory (tabela configuracoes)

ChaveDescrição
ad_servidorIP ou hostname do servidor LDAP
ad_portaPorta LDAP (padrão: 389)
ad_dominioFQDN do domínio (ex: empresa.com)
ad_basednBase de pesquisa (ex: DC=empresa,DC=com)
ad_usuario_svcLogin da conta de serviço para busca no AD
ad_senha_svcSenha da conta de serviço
Nota sobre ldapjs v3

O ldapjs v3 retorna atributos como arrays via entry.attributes[].values. O helper lerAtributo() em lib/ad.js normaliza esse comportamento garantindo compatibilidade. O bind é feito no formato UPN: login@dominio.

Rotas da API

6.1 Autenticação — routes/auth.js

POST/loginAutentica local (bcrypt) ou via AD (LDAP). Cria sessão, registra log. Retorna { sucesso, redirecionar }.
GET/logoutDestrói sessão e redireciona para /login.html.
GET/sessaoRetorna dados do usuário logado (nome, usuario, nivel, tipo). Usado pelo menu.js.

6.2 Portal e Configurações — routes/portal.js

GET/portalServe portal.html. Registra acesso no log.
GET/sistemasRetorna JSON com sistemas ativos ordenados por nome.
GET/configuracoesServe configuracoes.html. Requer nível admin.
GET/conf/:abaServe fragmento HTML de cada aba (lazy loading). Abas: usuarios-locais, usuarios-dominio, sistemas, telegram, email, seguranca, logs.
GET/logsRetorna últimos 50 registros de atividade. Requer admin.

6.3 Usuários e Sistemas (CRUD)

GET/api/usuariosLista todos os usuários locais.
POST/api/usuariosCriar usuário local com hash bcrypt.
PUT/api/usuarios/:idEditar nome, nível ou senha.
PATCH/api/usuarios/:id/statusAtivar / bloquear usuário.
DELETE/api/usuarios/:idExcluir usuário local.
GET/api/sistemasLista todos os sistemas (admin).
POST/api/sistemasCriar sistema.
PUT/api/sistemas/:idEditar sistema.
PATCH/api/sistemas/:id/statusAtivar / desativar sistema.
DELETE/api/sistemas/:idExcluir sistema.
POST/api/configuracoesSalvar configurações por grupo (telegram, email, seguranca, ad).

6.4 Active Directory — routes/portal.js

POST/api/ad/testarTesta conexão com o AD usando a conta de serviço configurada.
GET/api/ad/buscarLista todos os usuários ativos do AD (via LDAP). Usado pelo modal "Buscar no AD".
GET/api/ad/usuarios-portalLista usuários do AD já cadastrados na tabela usuarios_dominio.
POST/api/ad/usuarios/loteAdiciona múltiplos usuários do AD ao portal de uma vez.
PUT/api/ad/usuarios/:idAltera nível de acesso do usuário de domínio.
PATCH/api/ad/usuarios/:id/statusAtivar / bloquear usuário de domínio.
DELETE/api/ad/usuarios/:idRemover usuário de domínio do portal.

6.5 Agenda de Tarefas — routes/agenda.js

Todas as rotas requerem autenticação. Permissões: dono > edição > leitura.

GET/agendaServe a página HTML da Agenda de Tarefas.
GET/api/agenda/listasRetorna listas do usuário (próprias + compartilhadas com permissão e nome do dono).
POST/api/agenda/listasCriar nova lista. Retorna o objeto criado com permissao: 'dono'.
PUT/api/agenda/listas/:idEditar lista (somente dono).
DELETE/api/agenda/listas/:idExcluir lista em cascata: tarefas → categorias → membros → lista (somente dono).
GET/api/agenda/listas/:id/membrosListar membros da lista.
POST/api/agenda/listas/:id/membrosAdicionar ou atualizar membro (somente dono). Usa UPSERT.
DELETE/api/agenda/listas/:id/membros/:usuarioRemover membro (somente dono).
GET/api/agenda/listas/:id/tarefasListar tarefas (ordenadas por status, prioridade, prazo).
POST/api/agenda/listas/:id/tarefasCriar tarefa (requer edição ou dono).
PUT/api/agenda/tarefas/:idEditar tarefa completa (título, descrição, prazo, prioridade, status, categoria).
PATCH/api/agenda/tarefas/:id/statusAtualizar apenas o status (checkbox rápido).
DELETE/api/agenda/tarefas/:idExcluir tarefa.
GET/api/agenda/listas/:id/categoriasListar categorias da lista.
POST/api/agenda/listas/:id/categoriasCriar categoria com cor.
DELETE/api/agenda/categorias/:idExcluir categoria (desvincula tarefas antes de excluir).
GET/api/agenda/usuariosLista todos usuários ativos (locais + domínio) para o seletor de membros.

Página de Configurações (Abas)

Todas as 7 abas estão implementadas e funcionais:

  • ✅ Usuários Locais — Tabela com CRUD completo: criar, editar nome/nível/senha, bloquear/ativar, excluir
  • ✅ Domínio (AD) — Configuração LDAP, teste de conexão, lista de usuários com acesso, modal "Buscar no AD" com checkbox em lote
  • ✅ Sistemas — CRUD completo: criar, editar, ativar/desativar, excluir, ícone Font Awesome
  • ✅ Telegram — Configuração de bot (token + chat ID), toggle de notificações, botão de teste
  • ✅ Email (SMTP) — Configuração SMTP completa, toggle SSL, botão de teste
  • ✅ Segurança — Tempo de sessão, tentativas de login, bloqueio, política de senha
  • ✅ Logs — Tabela com filtros por usuário, ação, IP e período

Funcionamento do Lazy Loading

Cada aba faz um fetch() para /conf/{nome-da-aba} e injeta o HTML retornado no painel. Scripts dentro dos fragmentos são recriados dinamicamente via document.createElement('script'). Abas já carregadas ficam em memória.

Aba Domínio (AD) — Fluxo completo

  1. Admin preenche servidor, porta, domínio, Base DN, usuário e senha de serviço
  2. Clica "Salvar" → POST /api/configuracoes com grupo ad
  3. Clica "Testar conexão" → POST /api/ad/testar → bind LDAP com conta de serviço
  4. Clica "Buscar no AD" → GET /api/ad/buscar → lista todos os sAMAccountName ativos
  5. Admin seleciona usuários via checkbox, escolhe nível e clica "Adicionar selecionados"
  6. POST /api/ad/usuarios/lote insere na tabela usuarios_dominio
  7. Usuários adicionados podem logar com suas credenciais de rede

Categorias

Cada lista tem suas próprias categorias com nome e cor personalizada. Ao excluir uma categoria, as tarefas vinculadas ficam sem categoria (não são excluídas).

Compartilhamento

O dono pode adicionar membros ao projeto (usuários locais ou de domínio) com permissão de leitura ou edição. A alteração de permissão é feita inline no modal de membros sem necessidade de recarregar. Remoção de membro revoga o acesso imediatamente.

Como Executar o Projeto

Pré-requisitos

  • Node.js 18+ instalado
  • SQL Server acessível na rede
  • Arquivo .env configurado na raiz
  • (Opcional) Active Directory acessível na porta 389

Primeira execução

# 1. Instalar dependências (inclui ldapjs)
npm install

# 2. Criar .env com credenciais do banco (veja seção 4.3)

# 3. Iniciar o servidor (cria banco e tabelas automaticamente)
npm start

Atualizar banco em instalação existente

# Adiciona as novas tabelas (IF NOT EXISTS — seguro rodar novamente)
node criarBancoPortal.js

# Reiniciar o servidor
pm2 restart portal

Deploy de novos arquivos no servidor

# Copiar arquivos para o servidor Linux
scp portal.js kasuo@192.168.0.80:/var/www/html/wkl/Portal/portal.js
scp routes/agenda.js kasuo@192.168.0.80:/var/www/html/wkl/Portal/routes/agenda.js
