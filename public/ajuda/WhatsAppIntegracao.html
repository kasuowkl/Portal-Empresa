Integração WhatsApp — Guia Técnico
Como a integração foi construída, quais arquivos foram criados/alterados e como adicionar novos sistemas ao bot.

Visão técnica da integração

A integração foi construída em duas camadas: modificações no Portal WKL para aceitar chamadas de serviços externos, e a criação do serviço WhatsApp Bot que consome essas APIs.

O princípio central é simples: o Portal já tinha todas as regras de negócio (aprovações, usuários, permissões). A integração apenas reutilizou as APIs existentes, adicionando um mecanismo de autenticação para serviços externos e um novo endpoint de busca por número.

CamadaResponsabilidadeTecnologia
Portal WKL 3000 Regras de negócio, banco de dados, autenticação, APIs REST Node.js + Express + SQL Server
WhatsApp UI 3001 Conexão com WhatsApp, recepção e envio de mensagens Baileys (lib WhatsApp Web) + Socket.IO
WhatsApp Bot 3002 Interpretação de comandos, chamadas ao Portal, respostas Node.js + Express + Axios

Como a autenticação funciona

O Portal usa sessões de browser para autenticar usuários normais. Para o bot acessar as APIs sem uma sessão, foi criado um mecanismo de chave de API compartilhada.

O protocolo

O bot faz chamadas HTTP ao Portal enviando dois elementos:

  1. Header x-api-key: <chave> — prova que é um serviço autorizado
  2. Query param _whatsapp_login=<login> — informa qual usuário está agindo

Exemplo de chamada feita pelo bot para listar aprovações:

// services/portalApi.js — bot chama o Portal assim:
GET http://portal:3000/api/aprovacoes
  ?_whatsapp_login=joao.silva
  Headers: { 'x-api-key': 'wkl-whatsapp-2026' }

O Portal verifica a chave e, se válida, injeta o usuário informado na sessão temporária para processar a requisição normalmente — usando as mesmas regras de negócio do browser.

Middleware do Portal — verificarLogin.js

O arquivo middleware/verificarLogin.js foi atualizado para a versão 1.1.0, adicionando suporte a serviços externos.

portal/middleware/verificarLogin.js
Alterado — versão 1.1.0 — adicionado bloco de API key
function verificarLogin(req, res, next) {
  // 1. Sessão normal de browser — funciona como antes
  if (req.session && req.session.usuario) {
    return next();
  }

  // 2. NOVO — Acesso por serviço externo via API key
  const apiKey       = req.headers['x-api-key'];
  const serviceLogin = req.query._whatsapp_login || req.body?._whatsapp_login;
  const chaveCorreta = process.env.WHATSAPP_API_KEY;

  if (apiKey && chaveCorreta && apiKey === chaveCorreta && serviceLogin) {
    // Injeta usuário temporário na sessão para essa requisição
    req.session.usuario = { usuario: serviceLogin, nivel: 'usuario' };
    return next();
  }

  // 3. Sem sessão e sem API key válida — redireciona ao login
  req.session.redirectAfterLogin = req.originalUrl;
  res.redirect('/login.html');
}
Por que injetar na sessão? Toda a lógica do Portal (filtros de aprovação, permissões, logs) usa req.session.usuario. Injetando ali, o bot reutiliza 100% das regras existentes sem precisar duplicar código.

Endpoint de busca por número — /api/usuarios/por-whatsapp

Para que o bot identifique o usuário automaticamente pelo número do WhatsApp, foi criado um novo endpoint no Portal que não usa o middleware padrão (só valida a API key, sem precisar de _whatsapp_login).

portal/routes/portal.js
Alterado — novo endpoint GET /api/usuarios/por-whatsapp/:numero
// Endpoint exclusivo para o bot — só valida a API key
router.get('/api/usuarios/por-whatsapp/:numero', async (req, res) => {
  const apiKey = req.headers['x-api-key'];
  const chave  = process.env.WHATSAPP_API_KEY;
  if (!chave || !apiKey || apiKey !== chave)
    return res.status(401).json({ erro: 'Não autorizado.' });

  const numero = req.params.numero.replace(/\D/g, '');

  // Busca em usuarios locais E usuarios de domínio
  const resultado = await pool.request()
    .input('whatsapp', sql.VarChar, numero)
    .query(`
      SELECT TOP 1 usuario AS login, nome, nivel
      FROM usuarios WHERE whatsapp = @whatsapp AND ativo = 1
      UNION ALL
      SELECT TOP 1 login, nome, 'usuario' AS nivel
      FROM usuarios_dominio WHERE whatsapp = @whatsapp AND ativo = 1
    `);

  if (!resultado.recordset.length)
    return res.status(404).json({ erro: 'Não encontrado.' });

  res.json(resultado.recordset[0]);
  // Retorna: { login, nome, nivel }
});

Fluxo completo — Aprovações pelo WhatsApp

1
Usuário envia mensagem no WhatsApp

Ex: aprovar 42 — o WhatsApp entrega ao número da empresa.

2
WhatsApp UI (3001) recebe via Baileys

O evento messages.upsert do Baileys dispara. O serviço extrai jid, texto e nome e faz POST para o Bot.

Arquivo: services/whatsapp.js — função iniciarWhatsApp()

3
Bot (3002) recebe em POST /api/mensagem-recebida

Chama processarMensagem() do messageHandler.js.

4
Auto-login pelo número

Se a sessão não tem login, o bot extrai o número do JID (559291002823@s.whatsapp.net → 559291002823) e chama GET /api/usuarios/por-whatsapp/559291002823 no Portal com a API key. Recebe { login, nome, nivel }.

5
Interpretação do comando

messageHandler.js identifica o comando aprovar com ID 42 e chama responderAprovacao(42, 'joao.silva', 'Aprovado', null).

6
Bot chama o Portal

PUT /api/aprovacoes/42/responder com body { decisao: 'Aprovado', _whatsapp_login: 'joao.silva' } e header x-api-key. O Portal valida, injeta a sessão e executa a aprovação normalmente.

7
Bot envia resposta ao usuário

Faz POST para WhatsApp UI: /api/mensagens/enviar com o JID e a mensagem de confirmação. O WhatsApp UI entrega via Baileys.

Como integrar outro sistema ao bot

O bot foi projetado para ser extensível. Para integrar um novo módulo do Portal (ex: Chamados, Tarefas, Patrimônio), siga os 3 passos abaixo.

O padrão da integração é sempre o mesmo: Portal expõe API → portalApi.js cria função → messageHandler.js cria comando.

Passo 1 — Expor a API no Portal

O Portal precisa ter rotas REST que o bot possa chamar. As rotas já protegidas pelo verificarLogin funcionam automaticamente com a API key — não é necessário criar novas rotas, apenas garantir que aceitam o parâmetro _whatsapp_login.

Verificar se a rota já funciona

Teste via curl se a rota aceita a chave de API:

curl -H "x-api-key: wkl-whatsapp-2026" \
  "http://localhost:3000/api/sua-rota?_whatsapp_login=admin"

Se retornar dados (não redirecionar para login), a rota já está pronta.

Se precisar criar uma rota nova

Adicione em routes/portal.js usando o mesmo padrão das outras rotas:

// GET /api/chamados/meus — Lista chamados do usuário logado
router.get('/api/chamados/meus', verificarLogin, async (req, res) => {
  const pool  = req.app.locals.pool;
  const login = req.session.usuario.usuario; // vem da sessão (ou da injeção via API key)

  const resultado = await pool.request()
    .input('login', sql.VarChar, login)
    .query('SELECT id, titulo, status FROM chamados WHERE criado_por = @login AND excluido = 0');

  res.json({ chamados: resultado.recordset });
});
req.session.usuario.usuario funciona tanto para browser (sessão real) quanto para o bot (sessão injetada pelo middleware). Não é necessário código especial.

Passo 2 — Criar função em portalApi.js

Adicione uma função no arquivo services/portalApi.js do serviço WhatsApp para chamar a rota do Portal:

whatsApp/services/portalApi.js
Adicionar as novas funções no final, antes do module.exports
// ── Exemplo: Chamados ──────────────────────────────────

async function listarMeusChamados(login) {
  const { data } = await api.get('/api/chamados/meus', {
    params: { _whatsapp_login: login }
  });
  return data.chamados || [];
}

async function abrirChamado(login, titulo, descricao) {
  const { data } = await api.post('/api/chamados', {
    titulo,
    descricao,
    _whatsapp_login: login   // no body para POST
  }, {
    headers: { 'x-api-key': KEY }
  });
  return data;
}

// Adicionar no module.exports:
module.exports = {
  buscarContatosPortal,
  buscarUsuarioPorWhatsapp,
  listarPendentes,
  listarAprovacoes,
  listarTodasPendentes,
  detalharAprovacao,
  responderAprovacao,
  listarMeusChamados,   // ← novo
  abrirChamado,          // ← novo
};
Para chamadas GET, passe _whatsapp_login nos params. Para POST/PUT, passe no body. O middleware do Portal lê dos dois lugares.

Passo 3 — Adicionar comando em messageHandler.js

Importe as novas funções e adicione os blocos de comando no services/messageHandler.js:

whatsApp/services/messageHandler.js
Alterar o require e adicionar blocos de comando
// 1. Adicionar no require do topo:
const {
  listarPendentes, listarAprovacoes, detalharAprovacao, responderAprovacao,
  buscarUsuarioPorWhatsapp,
  listarMeusChamados,  // ← novo
  abrirChamado,         // ← novo
} = require('./portalApi');

// 2. Adicionar 'chamados' e 'abrir' na lista de comandos autenticados:
const comandosAutenticados = [
  '1', '2', '3', '4', '5', '6', '7',
  'pendentes', 'detalhar', 'aprovar', 'reprovar',
  'chamados', 'abrir',     // ← novos
];

// 3. Adicionar os blocos de comando antes do catch final:

// ══ Listar meus chamados ══════════════════════════
if (acao === 'chamados') {
  const lista = await listarMeusChamados(sessao.login);
  if (!lista.length) return reply('Você não tem chamados abertos.\n\nDigite *0* para o menu.');

  const linhas = lista.map((c, i) =>
    `${i + 1}. *#${c.id}* — ${c.titulo}\n   Status: ${c.status}`
  ).join('\n\n');

  return reply(`*Meus Chamados (${lista.length}):*\n\n${linhas}\n\nDigite *0* para o menu.`);
}

// ══ Abrir chamado ════════════════════════════════
if (acao === 'abrir') {
  // Espera: "abrir Título do chamado | Descrição"
  const texto    = partes.slice(1).join(' ');
  const [titulo, desc] = texto.split('|').map(s => s.trim());
  if (!titulo) return reply('Informe o título: *abrir Título | Descrição*');

  const res = await abrirChamado(sessao.login, titulo, desc || '');
  return reply(`✅ Chamado *#${res.id}* aberto com sucesso!\n\nDigite *chamados* para ver seus chamados.`);
}

Atualize também o menu

Adicione a nova opção ao texto do menu principal em services/botConfig.js (campo menuPrincipal) ou no painel admin em http://servidor:3002/admin.

Dica: Comece sempre pelo mais simples — listar registros. Depois evolua para criar, atualizar e fechar. Teste cada comando no WhatsApp antes de avançar para o próximo.
Padrão de nomenclatura recomendado para comandos:
Use verbos simples em português: chamados, abrir, fechar, tarefas, nova. Evite comandos longos — o usuário digita pelo celular.
